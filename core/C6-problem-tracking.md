# C6 — Problem Tracking

## Rule

Projects must track meaningful problems in a way that preserves status, impact, and resolution path.

A problem may be a defect, risk, limitation, incident, or unresolved question with practical impact.

## Why

Without consistent problem tracking, teams lose operational memory and cannot distinguish open issues from accepted constraints.

## Required minimum

- A stable identifier or discoverable reference
- A status
- An impact or severity signal
- A short description of the issue
- A link to the fix, mitigation, or decision when available

## Application guidance

- Keep status vocabulary small and clear
- Separate active problems from resolved ones
- Treat accepted limitations explicitly, not as silent abandonment
- Escalate high-impact unresolved items visibly

## Examples

Good:

- "Open / Fixing / Resolved / Deferred / Accepted"
- "High / Medium / Low impact" or equivalent

Bad:

- "done maybe"
- Issue known only in a chat thread with no durable reference

## Out of scope

- One issue tracker tool
- One severity taxonomy
- One mandatory archive folder structure
