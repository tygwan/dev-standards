# R3 — Finding Archival

**Strength**: 🔴 MUST
**Version**: 0.1.0

---

## Rule

When a data quality issue, design flaw, bug, or any non-trivial
discovery is made during development, it MUST be archived via the
**6-step process** in `docs/findings/YYYY-MM-DD-<ID>-<slug>/`.

### The 6 steps

1. **Preserve (보관)**
   - Create `docs/findings/YYYY-MM-DD-<ID>-<slug>/`
   - Add a reproducible `audit.<ext>` script that diagnoses the issue from raw data
   - Export concrete evidence (CSV, JSON, parquet) to `data/` subdirectory

2. **Organize (정리)**
   - Structure: `{README.md, audit.<ext>, data/, figures/}`
   - One folder per finding (never merge multiple issues)
   - Name components: ISO date + issue ID + human-readable slug

3. **Visualize (시각화)**
   - At least **one** visualization (chart, diagram, heatmap)
   - Saved as `figures/NN_descriptive_name.png` (or .svg)
   - Generated by `make_figures.<ext>` script for reproducibility

4. **Record (기록)**
   - `README.md` with the required 5-section format (see below)
   - Inline embedded figures
   - Cross-references to source files and related findings

5. **Portal update (포털 업데이트)**
   - Add row to `docs/PROJECT-JOURNAL.md §1 Quick Problem Index`
   - Add entry to `docs/PROJECT-JOURNAL.md §2 Timeline`
   - Add detailed section to `docs/PROJECT-JOURNAL.md §3 Findings`
   - If resolution needs a decision, add to §4 Decisions or §6 Open Questions

6. **Commit (커밋)**
   - Single atomic commit covering all of the above
   - Push immediately (do not leave findings uncommitted)
   - Commit message references the finding ID

### Required README 5-section format

```markdown
# YYYY-MM-DD — <ID> — <Title>

**Severity**: 🔴/🟠/🟡 (see R7)
**Status**: 🔄/🛠/✅/📋/🔭 (see R7)
**Discovered by**: <how it was found>
**Affects**: <which part of the project>

---

## 1. Finding
One paragraph: what was discovered, key numbers, first impression.

## 2. Evidence
### 2.1 Reproducible audit
  Command to run the audit script.
### 2.2 Data artifacts
  Description of each CSV/JSON file in data/.
### 2.3 Visualizations
  Embedded figures with explanations.

## 3. Analysis
### 3.1 Root cause
### 3.2 Impact
### 3.3 Related known issues

## 4. Resolution
### 4.1 Options considered (with trade-offs, see R8)
### 4.2 Selected approach
### 4.3 Action items (checkbox list)
### 4.4 Resolution commit (hash when resolved)

## 5. References
Source files, tests, related docs, external issues.
```

## Rationale

### Why a dedicated directory per finding

- **Co-location of artifacts**: audit script, data, figures, and README all
  in one place. No digging through separate folders.
- **Dated naming**: sorts chronologically by default, makes "when was this
  found?" trivial to answer.
- **Severity + status**: immediate visual scan of what's open vs resolved.

### Why reproducible audit scripts

Six months later, when you ask "wait, was the finding actually this bad?",
you don't want to re-derive everything. You want to run `audit.py` and see
the same numbers again. This makes findings **falsifiable** — either the
script still reproduces the result or something has changed.

### Why visualizations are required

Tables of numbers lose their impact over time. A chart makes the severity
*immediately obvious* when you re-open the finding months later. Even a
single bar chart is worth more than a CSV for human recall.

### Why portal update is step 5

The single portal (`PROJECT-JOURNAL.md`) is how future-you (or collaborators)
navigate the project. If findings get archived but the portal isn't updated,
they become invisible — effectively not-found.

### Why all 6 steps are atomic

If any step is skipped:
- **No preservation** → can't reproduce
- **No visualization** → severity lost over time
- **No record** → can't explain to others
- **No portal** → invisible
- **No commit** → lost when working tree changes

Splitting across commits makes it easy to leave half-done archives lying
around.

## Examples

### Good — a finding discovered during verification

```
docs/findings/2026-04-12-M1-classification-bug/
├── README.md                            # 5-section report
├── audit.py                             # Diagnosis script
├── make_figures.py                      # Visualization generator
├── data/
│   ├── confidence_breakdown.csv         # HIGH/LOW/BUG counts
│   ├── root_cause_attribution.csv       # Which bug caused which
│   └── affected_sample.csv              # Top 20 affected objects
└── figures/
    ├── 01_confidence_distribution.png   # Bar chart
    ├── 02_root_causes.png               # Attribution chart
    ├── 03_affected_patterns.png         # Top-N display
    └── 04_before_after.png              # Class distribution comparison
```

### Bad

```
docs/issues.md                # Single file for all issues
```

or

```
docs/findings/classification-bug.md   # No date, no structure, no evidence
```

## Enforcement

### When a finding IS required

- Data quality issue discovered during verification (even if small)
- Design flaw revealed during analysis
- Unexpected behavior in external dependencies
- Root-cause investigations for test failures
- Trade-off decisions that deferred a resolution
- Anything you might regret not recording in 6 months

### When a finding is NOT required

- Typos that are fixed in the same commit
- Tests that failed once and passed on retry (unless pattern emerges)
- Minor refactoring without issue
- Completed features without deferred concerns

### In code review

Before merging a PR that introduces or discovers an issue:

- [ ] Finding folder created with dated ID
- [ ] `audit.<ext>` script can be run by reviewer
- [ ] At least 1 visualization embedded in README
- [ ] README has all 5 sections filled
- [ ] PROJECT-JOURNAL.md updated (4 places minimum)
- [ ] Commit message references the finding ID
