# R9 — Provenance and Reproducibility

**Strength**: 🟡 SHOULD
**Version**: 0.1.0

---

## Rule

Projects SHOULD treat their external inputs, intermediate artifacts,
and claimed results as **traceable and reproducible**. Specifically:

1. **Pin external versions** explicitly
2. **Lineage columns** in processed data
3. **Reproducible audit scripts** for findings and claims
4. **Authoritative version identifiers** in output artifacts

### 9.1 Pin external versions

Any external input to the project (data snapshot, upstream tool,
specification) MUST have an explicit version identifier pinned
somewhere traceable:

- **Code constant** — e.g., `SNAPSHOT = "2026-04-07"` in a config module
- **Config file** — e.g., `external_tool_version: 1.4.0` in YAML/JSON
- **Lock file** — language-specific dependency lock

The pin MUST be documented in `docs/PROJECT-JOURNAL.md §5 External
Dependencies` with the rationale for the specific version.

### 9.2 Lineage columns in processed data

When the project transforms external data into its own format, the
output SHOULD include lineage metadata so consumers can trace back
to the source:

- **source_system** — which upstream tool produced the raw data
- **source_version** — pinned version of that tool or dataset
- **processed_at_utc** — ISO 8601 timestamp of the processing run
- **processing_rule** — name of the transformation applied
- **processing_rule_version** — version of the rule (for reproducibility)

The exact column names can vary by project; what matters is that the
lineage **can be reconstructed from the data itself**, not just from
external memory.

### 9.3 Reproducible audit scripts

Findings (R3) and decisions based on data (R4) SHOULD include an
**audit script** that can re-derive the reported numbers from raw
sources.

Required properties:
- **Deterministic** — same inputs → same outputs
- **Self-contained** — can be run by a new collaborator with minimal setup
- **Versioned** — lives in the finding folder, committed with the finding
- **Parameterized** — uses configuration constants, not hardcoded paths
  where possible

An audit script is the **falsifiability test** for the claims in the
finding. If someone disputes the finding, they can run the script.

### 9.4 Authoritative version in outputs

When the project produces deliverables (artifacts, reports, datasets),
those deliverables SHOULD embed their own version information:

- Dataset Parquet files → metadata columns `processed_at_utc`, `rule_version`
- Reports → front-matter or footer with generation date + source commit
- API responses → version field in JSON envelope

This enables "which version of the pipeline produced this?" to be
answered by inspecting the artifact.

## Rationale

### Why pin versions

External things drift:
- Upstream repos push new commits
- Datasets get refreshed
- APIs deprecate endpoints

Without pins, "this worked last week" becomes an unreproducible assertion.

### Why lineage columns

Without lineage, a processed dataset is a black box:
- "Why does this row say X?"
- "Which rule decided this?"
- "When was this generated?"

With lineage columns, every row carries its own provenance. A consumer
can filter `WHERE rule_version = 'v1'` to check if an old rule is still
being used.

### Why reproducible audit

Findings are assertions about data:
- "24% of X is wrong"
- "There are 997 affected rows"
- "The bug is in module Y"

Without reproducible audits, these assertions are just claims. With
them, they're testable. The difference is critical for:
- Communicating with stakeholders
- Receiving upstream fixes (they can verify the issue)
- Regression testing (re-run audit after fixes)

### Why authoritative version in outputs

When a consumer receives a dataset file, they often don't know:
- When was this generated?
- Which version of the pipeline made it?
- Which rules were applied?

Embedding the version in the output itself eliminates this ambiguity.
Pointing to "the repo's latest commit" is not enough — commits change.

## Examples

### Good — pinned version in config

```python
# config.py
SNAPSHOT: str = "2026-04-07"
EXTERNAL_TOOL_VERSION: str = "1.4.0"
DATA_RAW: Path = DATA_ROOT / "raw" / SNAPSHOT
```

And in the journal:

```markdown
## 5. External Dependencies

### <external-tool>

**Version pinned**: 1.4.0 (snapshot 2026-04-07)

**Why this version**: The 1.4.0 release is the last one before the
breaking API change in 2.0. We depend on the 1.x response format.
```

### Good — lineage columns

```python
# After processing, the output dataframe includes:
df["source_system"] = "external-tool"
df["source_version"] = "1.4.0"
df["processed_at_utc"] = datetime.now(timezone.utc).isoformat()
df["processing_rule"] = "standard_normalization"
df["processing_rule_version"] = "v1"
```

### Good — audit script in finding folder

```
docs/findings/2026-04-12-M1-slug/
├── README.md
├── audit.py                   # ← Reproduces all numbers in README
├── data/                      # Generated by audit.py
│   ├── breakdown.csv
│   └── samples.csv
└── figures/                   # Generated by make_figures.py
    └── 01_chart.png
```

The audit.py script:
```python
def main():
    df = pd.read_parquet(config.DATA_GOLD_TABLE)
    total = len(df)
    affected = (df["flag"] == "BUG").sum()
    pct = affected / total * 100

    print(f"Total rows: {total}")
    print(f"Affected:   {affected} ({pct:.1f}%)")

    # Export evidence
    breakdown.to_csv("data/breakdown.csv", index=False)
```

### Bad

```python
# No version pinning
df = pd.read_csv("/some/external/source.csv")

# No lineage
df_out = df.apply(some_transform)
df_out.to_parquet("output.parquet")

# No audit - just a claim in a doc
# "24% of rows are wrong, trust me"
```

## Enforcement

### At project start

- [ ] External dependencies pinned in config
- [ ] `PROJECT-JOURNAL.md §5` populated with version rationale

### During data processing

- [ ] Output datasets include lineage columns
- [ ] Transformation logic uses versioned constants (rule_version)

### For every finding

- [ ] Audit script accompanies the finding
- [ ] Script is deterministic and parameterized
- [ ] Running the script reproduces numbers in the README

### For every deliverable

- [ ] Version info embedded in the artifact itself
- [ ] Source commit referenced if possible

## Out of scope

- **Full bit-for-bit reproducibility** — R9 aims for logical
  reproducibility, not byte-level. Environment drift (library versions,
  OS) is not addressed here.
- **Secret management** — use your platform's tooling
- **Hash verification of external sources** — optional enhancement
  for projects with stricter integrity needs
